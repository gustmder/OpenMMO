# 맵 및 지형 시스템 설계 (Map & Terrain Design)

이 문서는 OnlineRPG의 거대 오픈월드 구현을 위한 지형 시스템과 맵 에디터의 기술적 사양을 정의한다.

## 1. 세계 규모 및 해상도 (World Scale)

*   **전체 월드 크기:** 32km x 32km (또는 64km x 32km 확장 가능)
    *   약 1,024 $km^2$ 규모의 심리스(Seamless) 오픈월드 지향.
*   **지형 해상도:** 1m/px (Grid-based)
    *   하이트맵(Heightmap)의 한 픽셀이 실제 게임 월드의 1m x 1m 면적을 담당한다.
    *   캐릭터의 이동과 상호작용은 이 1m 그리드를 기준으로 이루어진다.
    *   0.5m 해상도 대비 1m를 선택한 이유:
        *   0.5m로 올리면 타일당 버텍스 4배(4K→16K), 삼각형 4배(8K→32K), 저장 4배(8KB→32KB)로 비용이 급증한다.
        *   하이트맵은 XZ 좌표당 높이 값이 하나뿐이므로, 해상도를 올려도 수직 절벽이나 오버행(돌출 지형)은 표현할 수 없다.
        *   절벽, 동굴, 바위벽 등은 별도 3D 오브젝트 메쉬를 배치하여 표현한다.
*   **다층 지형 처리:**
    *   하이트맵은 XZ 좌표당 높이가 하나뿐이므로 다리, 2층 구조, 터널 등 수직으로 겹치는 지형은 표현할 수 없다.
    *   이러한 구조물은 별도 3D 오브젝트 메쉬로 배치하며, 하이트맵은 그 아래의 기반 지형(강바닥, 지면 등)을 담당한다.
    *   충돌 처리: 오브젝트에 "걸을 수 있는 높이" 속성을 부여하여, 플레이어가 오브젝트 위에 있으면 오브젝트 높이를, 아래에 있으면 하이트맵 높이를 사용한다.
*   **하이트맵 데이터 사양:**
    *   **데이터 타입:** uint16 (2바이트/픽셀)
    *   **해석:** `height = value * 0.1 - 500.0` (미터 단위)
    *   **높이 범위:** -500.0m ~ +6,053.5m (0.1m 정밀도)
    *   **해수면 기준:** value = 5000 → 0.0m
    *   **타일당 저장 크기:** 64 x 64 = 8,192바이트 (가장자리 메쉬는 인접 타일의 0번째 행/열에서 참조)

## 2. 타일링 및 로딩 시스템 (Tiling & Loading)

*   **타일 크기 (Tile Size):** 64m x 64m
    *   전체 월드는 $500 \times 500$ (32km 기준) 개의 타일로 분할 관리된다.
    *   클라이언트 렌더링 및 하이트 데이터 관리의 최소 단위.
*   **세그먼트 해상도:** 64 x 64 Quads
    *   타일 하나당 $64 \times 64$개의 격자 메쉬를 생성하여 1m 해상도를 유지한다.
*   **로딩 전략:** 3x3 동적 스트리밍
    *   플레이어가 위치한 중앙 타일을 포함하여 주변 3x3 (총 9개) 타일을 메모리에 로드한다.
    *   쿼터뷰(Isometric) 특성상 화면에 노출되는 타일은 최대 4개이므로, 3x3 로딩으로 충분한 시야와 심리스 이동을 보장한다.

## 3. 지형 표현 및 렌더링 (Terrain Rendering)

*   **4-Layer PBR Splatting:**
    *   하나의 RGBA Splat Map을 사용하여 4개의 텍스처 레이어를 혼합한다.
    *   **R:** 풀/바위 지형 (rocky_terrain_02, 기본 tile scale: 8.0)
    *   **G:** 자갈 (gravel_floor, 기본 tile scale: 6.0)
    *   **B:** 붉은 라테라이트 흙 (red_laterite_soil_stones, 기본 tile scale: 10.0)
    *   **A:** 눈 (snow_02, 기본 tile scale: 4.0)
    *   각 레이어의 tile scale은 에디터에서 실시간 조절 가능. 위 값은 기본값이다.
*   **리전 메타데이터:** 리전(16x16 타일 = 1km x 1km) 단위로 텍스처 구성과 tile scale을 JSON으로 관리한다.
    *   각 리전은 RGBA 채널에 서로 다른 텍스처를 할당할 수 있다 (예: 눈 리전, 사막 리전, 숲 리전).
    *   같은 리전 내 타일들은 동일한 4개 텍스처를 공유하고, 스플랫맵 가중치만 다르다.
    *   파일: `terrain/meta/r{±xx}_{±zz}.json`
    *   REST: `GET /api/terrain/meta/{±rx}/{±rz}`
    *   예시:
        ```json
        {
          "layers": [
            { "texture": "rocky_terrain_02_1k", "tileScale": 8.0 },
            { "texture": "gravel_floor_1k", "tileScale": 6.0 },
            { "texture": "red_laterite_soil_stones_1k", "tileScale": 10.0 },
            { "texture": "snow_02_1k", "tileScale": 4.0 }
          ]
        }
        ```
    *   리전 경계에서 텍스처 전환이 발생할 수 있다. 경계 블렌딩은 향후 확장으로 처리한다.
*   **PBR 데이터:** 각 레이어는 Albedo, Normal, ORM(AO, Roughness, Metallic) 데이터를 포함하여 실시간 조명에 반응한다.
*   **시각화 가이드 (Debug Grid):**
    *   **1m 그리드:** 검은색 선으로 표시 (편집 기준점).
    *   **64m 그리드:** 빨간색 선으로 표시 (타일 경계선).

## 4. 맵 에디터 핵심 기능 (Editor Features)

지형 편집기는 클라이언트 내부에 "에디터 모드"로 구현하여 실시간 피드백을 극대화한다.

1.  **지형 높이 조절 (Height Painter):**
    *   마우스 클릭/드래그로 지형의 높낮이(Y축)를 조절.
    *   브러시 크기 및 강도(Falloff) 지원.
2.  **지형 질감 칠하기 (Splat Painter):**
    *   4가지 텍스처 레이어를 브러시로 직접 칠하여 지형의 질감을 결정.
    *   RGBA 가중치 조절을 통한 부드러운 전환 처리.
3.  **오브젝트 배치 (Object Placement):**
    *   나무, 바위, 집 등 정적 오브젝트를 그리드 단위로 배치.
    *   타일당 JSON 파일로 저장한다.
        *   파일: `terrain/objects/r+00_+00/o_+0000_+0000.json`
        *   REST: `GET /api/terrain/objects/{±x}/{±z}`
        *   예시:
            ```json
            {
              "objects": [
                { "type": "oak_tree_01", "x": 12, "z": 34, "rot": 90 },
                { "type": "stone_house_01", "x": 5, "z": 10, "rot": 0, "scale": 1.5 }
              ]
            }
            ```
        *   `x`, `z`는 타일 내 로컬 셀 좌표 (0~63). `rot`은 Y축 회전(도). `scale`은 생략 시 1.0.
    *   **NPC 스폰 포인트:** 에디터에서 NPC 스폰 위치를 셀 단위로 배치한다. 타일 오브젝트 파일의 `npcs` 배열에 저장.
        *   스폰 데이터: `{ "type": "weapon_merchant", "x": 6, "z": 10, "rot": 180 }`
        *   NPC 타입 정의(행동, 대화, 상점 등)는 게임 콘텐츠 시스템에서 별도 관리.
    *   **밀집 장식 (잔디, 꽃 등):** 스플랫맵 가중치 기반으로 절차적(procedural) 생성한다. 별도 저장 불필요.
        *   예: R 채널(풀 지형) 가중치가 높은 셀에 잔디 인스턴스를 자동 배치.
    *   **오브젝트 타입 정의 (Object Definitions):** 타입별 메타데이터를 JSON 카탈로그로 관리한다.
        *   파일: `terrain/object_defs/{type_name}.json`
        *   REST: `GET /api/terrain/object_defs` (전체 카탈로그) 또는 `GET /api/terrain/object_defs/{type}`
        *   주요 필드:
            *   `mesh`: GLB 모델 경로
            *   `category`: 분류 (tree, building, structure, rock, decoration 등)
            *   `size`: 셀 단위 바운딩 `[x, z]`
            *   `walkBlock`: 지상 이동 차단 셀 목록 (오브젝트 로컬 좌표) → 배치 시 이동 플래그 맵에 자동 반영
            *   `layer`: 다층 구조물용. `walkHeight`(위를 걸을 수 있는 높이), `walkCells`(위에서 걸을 수 있는 셀 목록). 단층 오브젝트는 `null`.
            *   `interior`: 실내 진입 시 가시성 제어. 건물 GLB 하나에 모든 파트를 포함하되, 메쉬 이름으로 런타임 분리.
                *   `occluderMeshNames`: 카메라를 가리는 메쉬 이름 목록 (지붕, 앞벽 등). 진입 시 숨김/반투명 처리.
                *   `enterCells`: 출입구 셀 좌표 (오브젝트 로컬).
                *   `innerCells`: 내부 셀 좌표. 플레이어가 이 셀에 진입하면 occluder를 숨긴다.
                *   실내가 없는 오브젝트는 `null`.
        *   예시 (건물):
            ```json
            {
              "mesh": "models/objects/stone_house_01.glb",
              "category": "building",
              "size": [3, 3],
              "walkBlock": [[0,0],[1,0],[2,0],[0,2],[2,2]],
              "layer": null,
              "interior": {
                "occluderMeshNames": ["Roof", "WallFront"],
                "enterCells": [[1,0]],
                "innerCells": [[1,1]]
              }
            }
            ```
        *   예시 (다리):
            ```json
            {
              "mesh": "models/objects/wooden_bridge_01.glb",
              "category": "structure",
              "size": [2, 6],
              "walkBlock": [],
              "layer": { "walkHeight": 3.0, "walkCells": [[0,0],[1,0],[0,1],[1,1]] },
              "interior": null
            }
            ```
        *   정적 데이터이므로 클라이언트 시작 시 전체 로드하거나, 리전 진입 시 필요한 것만 lazy 로드.
4.  **데이터 저장 및 관리:**
    *   타일 데이터는 헤더 없는 raw 바이트 배열로 저장한다.
        *   **하이트맵:** uint16 리틀엔디안, row-major 순서 (`[z=0,x=0], [z=0,x=1], ... [z=63,x=63]`), 8,192바이트/타일
        *   **스플랫맵:** uint8 RGBA, row-major 순서, 16,384바이트/타일 (64x64x4)
    *   네트워크 전송 시 gzip 압축 적용 (평탄한 지형은 높은 압축률).
    *   **좌표계:** 월드 중앙을 원점 (0, 0)으로 하는 부호 있는 타일 좌표를 사용한다.
        *   중앙 지역부터 콘텐츠를 제작하고 사방으로 점진적 확장이 가능하다.
        *   32km 월드 기준 타일 좌표 범위: -250 ~ +249 (각 축)
    *   **파일 구조:** 타일당 1파일, 리전 디렉토리(16x16 타일 = 1km x 1km)로 분류.
        ```
        terrain/
          height/
            r+00_+00/              ← 리전 디렉토리 (원점 근처)
              h_+0000_+0000.bin    ← 타일 (월드 좌표 x=0, z=0)
              h_+0001_+0000.bin
              ...
            r-01_+00/              ← 음수 방향 리전
              h_-0012_+0005.bin
              ...
          splat/
            r+00_+00/
              s_+0000_+0000.bin
              ...
        ```
        *   파일명은 부호 있는 월드 타일 좌표를 사용 (`h_{±x}_{±z}.bin`).
        *   리전당 최대 256개 파일, 리전 디렉토리 수 약 1,000개 (32km 기준).
    *   **네트워크 API:**
        *   타일 데이터는 REST API로 제공한다. HTTP 캐싱(ETag, Cache-Control) 및 CDN 배포에 유리하다.
            *   `GET /api/terrain/height/{±x}/{±z}` → 하이트맵 바이너리 (gzip)
            *   `GET /api/terrain/splat/{±x}/{±z}` → 스플랫맵 바이너리 (gzip)
            *   `PUT /api/terrain/height/{±x}/{±z}` → 에디터에서 수정한 타일 저장
        *   타일 변경 알림은 WebSocket으로 전달하고, 수신한 클라이언트는 REST로 최신 데이터를 다시 받는다.

## 5. 지형 물리 및 이동 제한 (Terrain Physics)

*   **경사 제한:** 인접 셀 간 높이 차이가 1.0m (45도)를 초과하면 이동 불가로 판정한다.
    *   판정 기준: `abs(nextHeight - currentHeight) > 1.0`
    *   완만한 경사와 계단은 통과 가능, 절벽은 차단.
*   **경사면 이동 속도:**
    *   경사에 따라 셀 이동 시간을 조절한다. 오르막은 느리게, 내리막은 빠르게.
    *   속도 배율: `speedMultiplier = 1.0 - 0.5 * (heightDiff / 1.0)` (heightDiff = 다음 셀 높이 - 현재 셀 높이)
        *   오르막 (+1.0m): 0.5배, 평지 (0m): 1.0배, 내리막 (-1.0m): 1.5배 (또는 1.0 캡)
    *   서버 검증: 그리드 기반 이산 이동이므로, 셀 전환 간격(시간)이 예상 범위 내인지만 확인한다 (네트워크 지연 허용 마진 포함).
*   **이동 플래그 맵 (Walkability Map):**
    *   셀당 1바이트, 4비트로 방향별 이동 차단을 표현한다 (N/E/S/W 각 1비트).
    *   에디터에서 건물, 성벽 등 오브젝트를 배치하면 해당 셀들이 자동으로 이동 불가로 마킹된다.
    *   타일당 64 x 64 = 4,096바이트. 하이트맵/스플랫맵과 동일한 구조로 저장.
        *   파일: `terrain/walk/r+00_+00/w_+0000_+0000.bin`
        *   REST: `GET /api/terrain/walk/{±x}/{±z}`
    *   서버는 이 그리드 플래그만으로 이동 검증 및 패스파인딩(A*)을 수행한다 — 3D 충돌 판정 불필요.
    *   **다층 구조물의 이동 판정:** 이동 플래그 맵은 기본 지형(ground) 레이어만 담당한다.
        *   다리, 2층 등 다층 오브젝트는 자체 메타데이터로 이동 가능 영역과 높이를 정의한다.
        *   서버는 플레이어의 현재 레이어를 추적하여 ground → 이동 플래그 맵, 오브젝트 레이어 → 오브젝트 메타데이터로 각각 검증한다.
        *   NetHack 스타일의 랜덤 생성 다층 던전이 다수 존재할 수 있다. 던전 내부는 월드 하이트맵과 독립된 자체 맵 데이터(층별 이동 플래그, 높이, 오브젝트 배치)를 사용한다.
*   **서버 측 검증:** 경사 제한과 이동 플래그를 모두 서버가 판단한다 (치트 방지). 클라이언트는 시각적 피드백만 담당.
    *   **위반 대응:** 서버는 잘못된 이동을 즉시 거부하고 마지막 유효 위치를 클라이언트에 강제 전송한다 (보정).
        *   네트워크 지연, 클라이언트 버그 등 정상적인 원인이 있을 수 있으므로 즉시 밴하지 않는다.
        *   위반 횟수를 카운터로 누적하고 시간 경과에 따라 감쇠시킨다.
        *   단계적 대응: 보정만 → 경고 메시지 → 임시 킥 → 로그 기록 후 운영자 수동 검토.
*   **이동 프로토콜 (Movement Protocol):**
    *   **논리적 위치 (서버):** 셀 좌표 `(cellX, cellZ)` 정수 + 마지막 이동 시각.
    *   **시각적 위치 (클라이언트):** 자유로운 `(x, y, z)` 부동소수점. 셀 사이를 보간하여 부드러운 이동을 표현한다.
    *   **흐름:**
        1.  클라이언트에서 캐릭터가 셀 경계를 넘는 순간, `move(direction)` 패킷 전송 (N/E/S/W, 1바이트).
        2.  서버가 셀 단위로 검증 (이동 플래그, 경사, 속도 타이밍).
        3.  승인 시 서버의 논리적 위치 갱신 → 주변 플레이어에게 브로드캐스트.
        4.  클라이언트는 승인된 셀 간을 보간(interpolation)하여 3D 위치를 부드럽게 갱신. Y좌표는 하이트맵에서 보간.
    *   전투, 아이템 획득 등 게임플레이 판정은 모두 셀 좌표 기준으로 처리한다.

## 6. 월드맵 (World Map)

*   **이미지 생성:** 서버가 하이트맵과 스플랫맵 데이터를 기반으로 리전 단위 월드맵 이미지를 자동 생성한다.
    *   하이트맵 → 지형 윤곽 및 고도별 색상 (낮음=초록, 높음=갈색/흰색, 해수면 이하=파란색).
    *   스플랫맵 → 바이옴 대표 색상 블렌딩 (풀=초록, 자갈=회색, 흙=갈색, 눈=흰색).
    *   오브젝트 배치 → 주요 건물, 던전 입구 등을 아이콘으로 표시.
    *   해상도: 리전(1,024x1,024 셀)을 64x64px로 다운샘플. 32km 월드 전체 약 2,048x2,048px.
    *   생성 타이밍: 에디터에서 타일 저장 시 해당 리전의 월드맵 이미지를 자동 재생성.
*   **저장:**
    *   파일: `terrain/worldmap/wm_{±rx}_{±rz}.png`
    *   REST: `GET /api/terrain/worldmap/{±rx}/{±rz}`
*   **탐험 기반 공개 (Fog of War):**
    *   월드맵은 처음부터 전체가 공개되지 않는다. 플레이어별로 공개된 리전 집합을 서버가 관리한다.
    *   공개 방법:
        *   NPC에서 지도 아이템 구매 → 특정 리전(들) 공개.
        *   선택적: 플레이어가 리전에 진입하면 해당 리전 자동 공개.
    *   클라이언트는 공개된 리전의 월드맵 이미지만 표시하고, 미공개 리전은 안개 처리한다.
    *   현재 위치 마커는 항상 표시. 줌 인/아웃 및 스크롤 지원.

## 7. 성능 목표 (Performance Targets)

*   **프레임 레이트:**
    *   **권장 사양:** 60fps (RTX 4060, RX 7600 급)
    *   **최소 사양:** 30fps (GTX 1060, Intel Iris Xe 급)
*   **렌더링 버짓 (60fps = 16.6ms/프레임):**
    *   **지형:** 9타일 x ~8K 삼각형 = ~72K 삼각형, draw call 9회. 전체 버짓의 소량.
    *   **오브젝트:** 주요 비용. 동일 타입 오브젝트는 인스턴싱(Instancing)으로 draw call 최소화.
    *   **캐릭터:** 화면 내 동시 렌더 수 제한 (예: 최대 50명, 나머지는 네임택만 표시).
*   **낮은 사양 대응:**
    *   장식 오브젝트 밀도 조절 (잔디 등 절차적 데코 감소/제거).
    *   그림자 품질 및 해상도 단계 조절.
    *   캐릭터 동시 렌더 수 축소.

## 8. 향후 확장 계획

*   **LOD (Level of Detail):** 원거리 지형을 위한 저해상도 메쉬 시스템.
*   **수계 시스템 (Water System):** 강, 호수, 바다의 수위 및 흐름 표현.
*   **동적 환경:** 시간대 및 날씨 변화에 따른 지형 텍스처/조명 동기화.
